Redline Rewrite Plan

For the implementer (start here)

This document is the spec. It is a full rewrite, HTML-first. Read it top to bottom first; the Locked decisions, Anchoring & reconciliation model, Multiple documents, and Server & agent API sections are binding (the Data shapes section is an explicitly non-binding starting suggestion — see its note).

  • Build the new layout under src/ (core, formats/html, app, server, agent, client, shared); public/ = static assets only; client is Preact bundled by Bun.build to dist/ (gitignored).
  • Read the current code as reference, not to extend: src/state.ts (embedded state + quote/occurrence anchoring + source-preserving mutation), public/app-helpers.js (shared quote matching — the format-neutral seed of core anchoring), public/app.js (current vanilla client), and src/*.test.ts (the behavior spec to port).
  • Key changes from today's code: replace quote + occurrence with the selector model (quote + prefix/suffix + position); stop writing inline data-redline-anchor spans to disk (highlight at render instead); rename "resolve" to thread delete; address documents by docId. The embedded state block therefore gets a new schemaVersion.
  • Do not migrate old Redline files — ignore them. Remove the old src/*.ts + public/app.js at the final gate, after the new path passes the ported tests.
  • Project conventions (commit policy, etc.) are in your loaded memory, not here.

Product framing

Redline is a reviewing-and-commenting app, not a document owner. Someone else — the agent, or the user's own tooling — authors and maintains the file. Redline exists only for reviewing, commenting, and user↔agent collaboration on top of that file.

Three consequences follow, and they drive every decision below:

  1. State lives inside the one file. No sidecar. The reviewed file stays a single, fully portable artifact that carries its own comments wherever it goes.
  2. Redline disturbs the content as little as physically possible. The only bytes it ever writes are its embedded review-state block (and, at most, a one-line discovery marker — see Embedded state block). It never reformats or restructures the document.
  3. Redline is a participant, never a gatekeeper. The file on disk is the shared source of truth. The agent may edit it directly at any time, with or without Redline running. Redline reconciles; it does not lock the file or become the sole editor (it is not Google Docs).

The rewrite exists to make these true before the project is open-sourced and before files in the wild create a migration burden. Old Redline-authored files are explicitly not migrated — we can ignore them and move on. This is the cheapest this refactor will ever be.

Roadmap this unlocks

  • More commentable formats (Markdown, code, etc.) — HTML is the first adapter, not the architecture.
  • Simultaneous editing — at least one user + one agent at once (agent responding while the user comments), more later.
  • Self-contained downloadable binary — clone-free, just-run distribution.
  • Open source — a clean, contributable core with a documented adapter contract.

Locked decisions

These were settled in design discussion and are not open for re-litigation in implementation:

  1. Embedded state, no sidecar. Review state is serialized into one block inside the file (HTML: <script type="application/json">; Markdown: a trailing HTML comment; code: a language comment). This block (plus, at most, a one-line discovery marker) is the only region Redline writes.
  2. No inline anchor markers on disk. Today's <span data-redline-anchor> wrappers are dropped. Highlights are rendered at view time, not persisted. The saved file's content is byte-identical except for the single state block.
  3. Fuzzy, context-aware anchoring. Anchors are stored as redundant selectors (quote + prefix/suffix context + position hint) and resolved with a fallback cascade, not exact-match-or-die. (Full model below.) This replaces the current quote + occurrence scheme.
  4. Redline owns reconciliation, automatically. The agent stays anchor-blind: it edits text, Redline re-resolves anchors on load / external change / render.
  5. Region ownership. Content = owned by the document's author (agent directly, or user via the later text-edit feature). State block = owned by Redline. Concurrent writers to different regions never clobber each other.
  6. User content editing: A now, B later.
    • A (now): user is a commenter; the agent owns all content. Redline never mutates content.
    • B (later, additive): re-introduce the "change a word/phrase" affordance as a locate-and-replace text op (reusing the anchor locate machinery), not as a whole-document reserialize.
    • The current whole-document serialize + PUT /api/document content path is removed regardless — it fights both minimal-disturbance and multi-format.
  7. Optional live coordination, never required. When a Redline server is already running for a file (tracked via the server registry), the agent CLI may route ops through it for presence and atomic anchor updates. With no server running, the CLI writes the file directly. The direct-write path is the floor; coordination is an optimization on top.
  8. Client leaves public/. Source lives in src/client; build output goes to a dedicated dir and is embeddable into the binary. public/ holds static assets only (fonts, favicon). No app code or built bundles under public/.
  9. Multiple documents per server (first-class). One server instance holds many open documents at once — keyed internally by canonical file path, addressed over the API by an opaque docId. Routes, events, and assets are document-scoped. Single-document is just the one-entry case. (Cheap because review state lives in the files; see Multiple documents.)

Non-goals / deferred

  • Sidecar state — excluded by the product framing, not just out of scope.
  • CRDT / OT same-region co-editing — fuzzy re-anchoring + graceful orphaning is the v1 concurrency floor. Sticky CRDT positions are a later investment, only if true character-level co-editing on the same text is wanted.
  • User content editing (B) — designed-for but not built in this rewrite.
  • Migration of old Redline files — not supported; not built.
  • Non-embeddable formats — every supported format must be able to carry its own state block in a comment/ignored region.

Architecture overview

Top-level subsystems:

  • src/core — format-neutral review model: threads, messages, authors/body normalization, the anchor selector model and fuzzy resolution cascade, and reconciliation. No knowledge of HTML, files, or transport.
  • src/formats/<id> — per-format adapters. html first. Each adapter knows how to extract anchoring text, render for viewing, and read/write only the embedded state block.
  • src/app — use-case commands composed from core + adapter: open/close document, create comment, reply, edit message, delete thread, re-anchor, apply agent update, reconcile-on-change. Each command is per-document; the server's session map supplies the document context.
  • src/server — Bun HTTP server holding a Map<docId, DocumentSession> (per-document path, version, file watcher, SSE subscribers); doc-addressed route handlers, per-document SSE channels, the server registry (lists a server's open docs), static serving of the built client + per-document assets, and the file picker.
  • src/agent — CLI parser and command adapters; routes through a live server when present (via registry), else applies directly.
  • src/client — the review UI (app shell + imperative iframe document viewer). Built to a dedicated output dir, not public/.
  • src/shared — DTOs shared across server / client / agent boundaries.

Anchoring & reconciliation model (the crux)

Selector schema

Each thread stores an anchor as a set of redundant selectors, captured at comment-creation time, inside the state block:

anchor: {
  quote:    string   // exact selected text (normalized)
  prefix:   string   // ~30–50 chars/words immediately before the quote
  suffix:   string   // ~30–50 chars/words immediately after the quote
  posStart: number   // approximate char offset in canonical text (hint only)
  posEnd:   number
}

prefix/suffix replace the old 1-based occurrence index — context disambiguates duplicates and survives reordering, where an occurrence count breaks the moment earlier identical strings are added or removed.

All three text fields (quote, prefix, suffix) are over the adapter's canonical text (extractText) — never the raw markup. For HTML that's the rendered text content with tags stripped.

How selectors are captured. On comment creation (and re-anchor) the client captures the full selector set — quote plus surrounding prefix/suffix and approximate posStart/posEnd — from the live selection in the rendered text layer, and sends all of it; the server does not reconstruct selectors from a bare quote. occurrence (in the CLI / agent API) is only a transient disambiguation input for choosing which instance of a repeated quote an agent means at capture time — it is resolved to concrete selectors immediately and is never persisted in the state block.

This is established prior art, not a bespoke scheme. The most relatable example is GitHub PR review comments — anchored to a line plus surrounding diff context, and marked "Outdated" when the code shifts (exactly orphaned / needs-review). Google Docs and Word reflow comments the same way, keeping a comment when its target text is deleted. Formally it is the W3C Web Annotation Data Model (TextQuoteSelector = exact + prefix + suffix, plus TextPositionSelector = start/end), with fuzzy matching as shipped by Hypothes.is (diff-match-patch). A proven model, adopted — not invented.

Resolution cascade

Resolved fresh against the current canonical document text (never against a persisted position):

  1. Exact quote + prefix/suffix near posStart → fast path, common case.
  2. On exact miss, fuzzy match the quote (approximate string match with a similarity threshold), scoring candidates by surrounding prefix/suffix agreement, biased toward posStart.
  3. Best candidate above the confidence threshold wins; otherwise the anchor is orphaned.

A small fuzzy text-matcher (diff-match-patch–style) is vendored into core to keep dependencies light.

Canonical-text parity (resolution ↔ highlight)

The server resolves anchors against the adapter's extractText (canonical text); the client paints highlights by re-finding the resolved quote in the rendered DOM's text layer. These must agree, so the text normalization is a single shared contract — whitespace collapsed, case-insensitive, script/style/hidden content excluded, block boundaries respected — used by both sides. The seed is today's public/app-helpers.js quote matcher, which already runs in both browser and server; keep that shared. Consequences: character offsets are server-side hints only, never shipped to the DOM as positions (the client matches by normalized text, not offset); and a quote the server resolves but the client cannot uniquely locate in the rendered DOM is shown as needs-review, never silently mis-highlighted.

Three states (never silently lost, never silently moved)

  • Anchored — confident match; highlight shown normally.
  • Needs review — fuzzy match below high confidence; shown but flagged, so a low-confidence re-anchor is never silently treated as correct.
  • Orphaned — no match above threshold; the comment stays in the thread list with its last-known quote/context in an "unanchored" section. Nothing is ever dropped.

Behavior under edits

External editOutcome
Edit elsewhere in the docQuote + context still match; resolution follows the shift.
Small edit inside the quoted textExact fails; fuzzy + context re-anchors, now covering the edited text.
Wholesale rewrite of the regionOrphaned and surfaced, not lost.

Ownership & timing

  • Redline (server) reconciles automatically on file open, on detected external change, and at render. Because highlights resolve at render time, there is no persisted anchor position that can be stale-and-wrong; the position hint only biases fuzzy search.
  • Reconciliation touches only the anchor (where a comment points) — never the comment body or thread, which are owned by their author.
  • Self-healing (lazy): a successful resolution refreshes the stored prefix/suffix/pos* hints so anchors track the document over many edits — but persisted lazily (on the next state write or on idle), never as a write that races the editing agent. Resolution is recomputed at render regardless, so stale hints only mildly affect fuzzy accuracy, never correctness.
  • Policy (default): when the quoted text itself changed slightly, let quote follow to the new text while recording that it changed (auditable), rather than freezing the original.
  • Orphan recovery: human re-selects the new text to re-attach (one gesture); or, when coordinating through a live server, an agent edit op carries the anchor id to re-point atomically (the Model-2 optimization).

Agent re-anchoring tools

Automatic reconciliation handles most edits by itself; the agent intervenes only for what it can't. The surface is two commands + one report, all read-only or state-block-only — the agent never writes content through Redline.

Reconcile (automatic, lossless, idempotent). Whenever Redline loads or renders the file it re-resolves every anchor against the current text and classifies each as anchored, needs-review, or orphaned. This is pure resolution — it never deletes or moves a comment, and re-running it yields the same result. Edits elsewhere, and small edits inside the quoted text, re-anchor silently with no agent action.

  1. List anchors (read). Returns every thread with its current resolution: state, the resolved character range in the canonical text (when matched), and the stored quote/context. The agent reads it before editing to see which comments sit in the region it is about to change, and after editing to see what moved. (Extends today's comments index with the resolved position.)
  2. Re-anchor (state-block write). reanchor <thread-id> --quote "<new text>" [--occurrence N] — points one comment at new text when reconcile couldn't (orphaned) or matched with low confidence (needs-review). For a whole-document rewrite, re-point many at once by submitting them as a batch via apply. Writes only the state block; never touches content.

Reconcile report. The output of List anchors is the report: the set of orphaned / needs-review threads the agent should fix. It is a queryable view of current state, not a one-shot event — ask any time and get the truth for the file as it is now.

The agent loop: (1) list anchors → note which sit in the region about to change; (2) edit the document text directly in the file; (3) list anchors again → read the orphaned / needs-review leftovers; (4) re-anchor only those (single or bulk).

There is no race between automatic reconcile and explicit re-anchor: reconcile is lossless and idempotent, every state-block write is serialized by Redline, and an orphaned classification is not final — re-anchor is simply how the agent supplies the answer reconcile couldn't find. A fuzzy match below the high-confidence threshold is demoted to needs-review rather than silently accepted, so a comment never quietly attaches to the wrong place.

The human's recovery path for the same leftovers is the Unanchored UI (see Client). The optional atomic path — handing Redline the new document plus re-anchors in one call — is deferred; the direct-edit-then-reconcile loop above is the floor.

Concurrency & ownership model

  • The file is the shared source of truth at rest. Multiple writers are expected.
  • Content region — written by the agent directly (any time, server or not) and, later, by the user's text-op edits. State block — written only by Redline.
  • Merge rule on any external content change: take their content, keep my block, re-resolve anchors. Because the two sides edit disjoint regions, the common case needs no locking.
  • The only genuinely conflicting case — two writers editing the same content region at the same instant — uses last-write-wins + a visible "document changed, reloaded" notice to start.
  • Live coordination (optional): when a server owns the file (registry), agent ops can route through it for instant appearance + atomic anchor updates. Floor behavior without a server is unchanged: direct file write + reconcile-on-reload.
  • Multiple clients are supported by design. Several clients (the user's browser + the agent, more later) connect to one server; each sends granular ops and receives others' over the event stream. Because comment ops mutate only the state block and the server serializes them, concurrent comment creation is just ordered writes — no clobbering. Designed-for, not hardened in this HTML-first rewrite.
  • Write safety, and accepted clobber. Redline writes atomically (temp file + rename) and, immediately before writing, re-reads and re-applies only its state block onto the current content — so it never overwrites the agent's edits. It persists on idle / piggybacked on real comment ops, never racing the editor. Filesystem locks are not used: OS file locks are advisory and an external agent won't honor them. If the agent rewrites the file and drops the state block, those comments are lost — this clobber risk is accepted as the price of letting an external agent be the primary writer. Redline keeps no on-disk recovery journal; nothing lives outside the one file.
  • Two Redline writers → CAS, not clobber. Registry routing makes one server the writer in the common case, but a stale/missing registry entry, a second server, or a direct CLI write can still race the state block. Guard every state-block write with an optimistic version/hash check against the current on-disk block: on mismatch, re-read and merge (threads are keyed by id, so independent adds/edits merge cleanly) or return a conflict. This is distinct from the accepted clobber above — that's the agent dropping the block via a content rewrite; this stops two Redline writers from losing each other's comments.

Format adapter contract

Minimal, because Redline never mutates content beyond the state block:

interface ReviewFormatAdapter {
  id: string
  label: string
  extensions: string[]
  canOpen(path): boolean

  extractText(raw): string          // canonical text for anchoring
  render(raw): RenderedView         // viewable HTML for the iframe (HTML: sanitized passthrough; MD: md→html; code: highlighted)
  readState(raw): EmbeddedState | null   // parse the embedded block
  writeState(raw, state): string         // inject/replace/remove ONLY the block; touch nothing else
}
  • Highlighting is client-side decoration over the rendered DOM: the server sends resolved anchors, the client locates and highlights the text in the rendered view (format-neutral text matching). No offset-mapping in the adapter.
  • This rewrite implements html only. The adapter interface is the extension seam, but no second format is built here. Unsupported formats return a clear "format not supported yet" error.
  • The seam is designed so later formats drop in as adapters — Markdown next, possibly DOCX after — without changing core. To keep that path open, the core must not assume HTML's source ≈ rendered property: it anchors against the adapter's extractText (the rendered text layer), so a format whose source differs from its rendered form (Markdown's **bold** → bold) still works. Beyond that, no format-specific work is in scope now.
  • Rendering is security-sensitive. Reviewed content is untrusted (AI- or third-party-authored). render() output is shown in a locked-down sandboxed iframe: scripts disabled, no inline event handlers, no form submission or navigation, external resource loading restricted, and no same-origin access to Redline's own API. Define the sandbox flags + sanitizer policy explicitly; the current app already sandboxes the iframe — keep at least that posture.

Embedded state block

The on-disk persistence rules (the current code has rich behavior here — read src/state.ts and keep these as golden rules):

  • Placement. One block per file. HTML: a <script type="application/json" id="redline-state"> in <head>. Markdown: a trailing HTML comment. Code: a language comment.
  • Escaping. JSON-escape < (to its unicode escape) so the payload can't break out of the host <script> / comment (current code does this — keep it).
  • Empty state. When the last thread is deleted, remove the block entirely — don't leave an empty shell.
  • Discovery + agent guidance (required). Redline stamps a one-line, human/agent-readable marker so an agent that later opens the file can self-orient. This is essential to the core flow: an agent writes a doc → the user opens it in Redline and leaves comments → the user tells the agent to "review" → a fresh agent opens the file and must know it's a Redline doc and what to do. HTML: <meta name="redline-agent-guide" content="Redline review document. Agents: use the redline-review skill; review state is in the #redline-state block."> in <head>; other formats: the same instruction in a leading comment. It points at the redline-review skill and the state block. Stamped when Redline first persists to the file (alongside the state block). This marker is the only thing Redline writes besides the state block — so don't claim "only the state block." (Note: the current marker text mentions data-redline-anchor spans — drop that, since spans are gone.)
  • Schema + robustness. Carry schemaVersion; ignore/reject unknown versions rather than silently corrupting. A malformed/unparseable block → treat as "no review state" and surface a warning; never overwrite a file whose block you couldn't parse without making that explicit.

Client

  • App shell (comment rail, active thread, composer, replies/editing, topbar, notices, save/conflict state, server-event refresh) owns review state.
  • Landing / root URL. / serves the app shell, which branches on GET /api/docs: no docs open → an empty state inviting you to open a file (built-in file browser + recent files); one or more open → the document switcher, defaulting to the most-recently-active doc (if exactly one is open, show it directly). A specific document is deep-linked as /?doc=<docId>; because docId is ephemeral, a stale link falls back to the switcher / open-by-path rather than erroring.
  • The iframe document viewer stays imperative behind a small controller (load rendered content, track selection, apply highlights, scroll alignment). React/UI does not model the reviewed document DOM.
  • No content-edit surface in A (no contenteditable, no beforeinput filtering, no dirty/autosave/whole-doc serialize). These are removed; B reintroduces editing as a content-edit op later.
  • Built with Preact (~3KB, React-style components + hooks) — chosen over React for footprint and the single-binary goal, with a near-mechanical upgrade path to React via preact/compat if ever needed. Optional @preact/signals for fine-grained, SSE-driven updates without re-rendering the whole rail. Bundled with Bun's built-in Bun.build (JSX jsxImportSource → preact); no extra build tooling.
  • Unanchored UI (orphan recovery). An "Unanchored" section in the rail lists orphaned and needs-review threads with their last-known quote; selecting one lets the user highlight new text in the doc and Re-attach to selection (one gesture). A "needs review" (low-confidence fuzzy) thread gets a badge to confirm or re-point. Prior art: Hypothes.is's "Orphaned" tab; Google Docs / Word retaining comments whose anchored text was deleted.

Project layout, build, and binary

  • src/ — all source (core, formats, app, server, agent, client, shared).
  • public/ — static assets only: fonts/, fonts.css, favicon.svg.
  • Build output — a dedicated dir (e.g. dist/), not under public/; embeddable into the binary.
  • index.html shell is authored under src/client and emitted by the build, not hand-served from public/.
  • Binary: bun build --compile produces a single executable with the client bundle and fonts embedded as assets (no runtime build step, no public/ read at runtime).
  • Nothing generated is committed. Both dist/ and the compiled binary are gitignored. CI builds them; per-platform binaries ship as GitHub Release assets, not in the tree. The server runs TypeScript directly (no build); only the client needs bundling, and dev can build-on-the-fly / watch.

File open (portable, server-side)

Path resolution lives server-side — the browser sandbox is irrelevant because Redline is a local server with full filesystem access; the client only ever sends/receives a path string. A previous attempt at a native picker wasn't great, so the portable approach is the base and native pickers are optional polish. Keep these behind one FilePicker interface so we can swap implementations (or return to a system-specific picker) without touching callers:

  • Primary — launch / CLI arg: redline ./doc.html opens directly; pair with a recent-files list.
  • Universal fallback — built-in file browser: the server lists directories/files and serves a small navigator UI; works on every OS/browser, no native deps, even headless/remote. This is the default for a double-clickable binary with no terminal.
  • Optional polish — native dialog per OS: keep osascript on macOS; Linux/Windows native dialogs are flaky/often-absent, so they're an opt-in implementation of the same FilePicker interface, not the foundation.
  • Approaches that rely on the browser surfacing a real path, or on uploading file bytes, are rejected — the latter breaks "edit the file in place on disk."

Multiple documents

A single server instance holds many open documents at once — first-class, not one process per file. This is cheap precisely because review state lives in the files: the server holds only ephemeral per-document machinery, never authoritative data.

  • Session map. The server keeps a Map<docId, DocumentSession>; each session owns that document's canonical path, version, file watcher, and SSE subscribers. There is no module-level "current document".
  • Addressing. Keyed internally by canonical absolute path (opening the same file twice returns the same session) and exposed over the API by an opaque docId from open — for clean, stable route keys (a raw path makes ugly, encoding-sensitive URLs and couples routes to the filesystem). Paths are not secret: they appear in the file browser, the docs list, and info — the opacity is URL/routing hygiene, not path-hiding. The agent CLI keeps taking a file path and resolves it to a session server-side. To get the docId from a path explicitly, an agent uses POST /api/docs {path} (open-or-return-existing → docId), the CLI's redline docid / redline info, or GET /api/docs?path= for an already-open doc.
  • docId scheme. Random and ephemeral — doc_ + three words drawn (via crypto.getRandomValues) from a curated, profanity-screened wordlist, e.g. doc_brave-otter-canyon. Recognizable in URLs and logs; zero dependency. Collisions are resolved by regenerating against the docId → path index, so correctness never depends on entropy (three words is ample headroom). Ids are not stable across server restarts — by design (see next).
  • Durable identity = the file path. The docId is an opaque, ephemeral session handle, not an identity. The durable reference is the file path (the file carries its own state and survives restarts for free). Clients and agents must always resolve path→docId and never persist a docId expecting it to outlive the session. The ephemerality is deliberate: ids visibly change across restarts, which stops anything from silently depending on their stability and keeps the decision reversible — stability (a hash, or a persisted index) can be added later if shareable deep links ever become a feature, without breaking a promise.
  • Doc-scoped everything. State, events, assets, comments, and anchors are addressed under /api/docs/:docId/…; each document has its own SSE channel and its own watcher, with an open/close lifecycle.
  • Registry. A server's registry entry lists all its open documents, so an agent discovers which server hosts a given file. If two live servers claim the same canonical path, the CLI picks deterministically (e.g. most-recently-started) and warns.
  • Switcher updates. Doc-scoped SSE can't announce a document a client hasn't subscribed to yet, so the document switcher learns about opens/closes from a server-level signal — a small server-level event stream or polling GET /api/docs.
  • Lifecycle. Define watcher behavior for rename/delete (treat as external change, then close), close while clients are connected (notify, then the doc routes 404), and reopening the same canonical path (same session; a fresh docId after a restart). A request with a stale docId returns "unknown doc — re-resolve by path," never a wrong document.
  • Scope. Server + API are built for real N in this rewrite; the client starts with a minimal document switcher (open / switch / close), with richer multi-document UX (tabs, side-by-side) as later polish.

Data shapes

These are a starting suggestion, not a spec — shapes derived from the discussion and review that produced this plan, meant to save you from inventing them cold. Treat them as a sensible default and deviate wherever the implementation has a better answer; the wire format is ours to set. The one thing to hold constant: whatever shapes you land on, keep them consistent across server / client / agent (they live in src/shared). Thread/message id format (thread_…, message_…) with validation, and author defaults (agent paths default to "AI", the generic reply path to "User"), follow the current code — preserve them.

// ---- core model ----
interface Message { id: string; author: string; body: string; createdAt: string; updatedAt?: string }
interface AnchorSelectors {
  quote: string;     // exact selected text, normalized, over CANONICAL TEXT (not markup)
  prefix: string;    // context before the quote (see "context window" knob)
  suffix: string;    // context after
  posStart: number;  // approx char offset in canonical text — a hint only
  posEnd: number;
  originalQuote?: string; // set when self-healing let `quote` follow an edit — keeps the change auditable
}
interface Thread {
  id: string;               // "thread_..."
  anchor?: AnchorSelectors; // omitted/null = document-level (general) comment, not tied to a quote
  author: string;
  createdAt: string;
  updatedAt: string;
  messages: Message[];      // [0] = original comment; rest = replies
}

// ---- embedded state block (the ONLY bytes Redline writes into the file) ----
interface EmbeddedState { schemaVersion: number; updatedAt: string; threads: Thread[] }

// ---- reconcile report (List anchors) ----
type AnchorState = "anchored" | "needs-review" | "orphaned";
interface AnchorStatus {
  threadId: string;
  state: AnchorState;
  range?: { start: number; end: number };  // resolved position when matched
  confidence?: number;                     // 0..1 for fuzzy matches
  quote: string;                           // last-known quote, for display
}

// ---- agent batch (apply / POST .../agent/update) — NO content field ----
interface AgentUpdate {
  comments?: { quote: string; body: string; author?: string; occurrence?: number }[];
  replies?:  { threadId: string; body: string; author?: string }[];
  edits?:    { threadId: string; messageId: string; body: string }[];
  deleteThreads?: string[];
  deleteReplies?: { threadId: string; messageId: string }[];
  reanchors?: { threadId: string; quote: string; occurrence?: number }[];
}

// ---- server ----
interface DocumentSession { docId: string; path: string; format: string; version: string } // + watcher, subscribers
// POST /api/docs            -> { docId, path, format, version }
// GET  /api/docs            -> { docs: DocumentSession[] }
// GET  /api/docs/:id/state  -> { docId, format, version, renderedHtml, threads: Thread[], anchors: AnchorStatus[] }

Server & agent API

Two clients use this surface: the browser UI and agents (via CLI or HTTP). Principles: we are free to redesign the wire format (old files/clients are not preserved); there is no content-write endpoint (the agent owns and edits content directly; the API only reads content and writes the comment state block); documents are addressed by an opaque docId under /api/docs/:docId/, with the server mapping docId ↔ canonical path; and every CLI command auto-discovers a running server via the registry (~/.local/state/redline/servers/), routes through it, and resolves the file path to a docId — falling back to operating on the file directly when no server runs. All responses carry format and version. Write endpoints accept an optional expectedVersion (the version the client last saw); a mismatch returns 409 with the current state so the client can rebase, rather than clobbering. Errors use conventional status codes with a { error } body (e.g. 404 for an unknown docId → "re-resolve by path"). CLI is the redline binary; in dev, bun src/agent.ts <command> / bun src/server.ts.

Deployment surfaces

The local-server design keeps these open. None are built now; none are precluded. Note that review state lives in the file, so the server is nearly stateless today — what varies between targets is the filesystem and the realtime transport, not where comments are stored.

  • Local binary (primary). A compiled executable; files on the user's own disk.
  • VPS / container (Fly, Render, a plain box). Runs as-is — a long-lived Bun HTTP server. Two shifts: the file picker browses server-side files, and a public deployment needs auth / multi-tenancy (out of scope, not precluded).
  • Electron desktop app. Good fit: run the compiled server as a sidecar (the single-binary work provides it) or port to Node, host the client in a BrowserWindow, and implement the FilePicker via Electron's native dialog.showOpenDialog.
  • Pure serverless (e.g. Vercel functions): awkward, not impossible. Review state is in the file, so this isn't blocked by in-memory state. What serverless lacks is (a) a persistent local filesystem — Redline edits a file on disk, and serverless disks are ephemeral, so the doc would have to live in durable storage the function reads/writes; and (b) long-lived connections for realtime push (SSE/WebSocket), so you'd fall back to polling. The only genuinely stateful piece is the future in-memory multiplayer model, which isn't built yet. Drop realtime and host the doc on durable storage and today's embedded model could run serverless.

Cutover approach

  • Not a big-bang. The existing test suite (state.test.ts, server.test.ts, agent.test.ts, app-helpers.test.ts) is the behavior spec — port the still-relevant cases (comment CRUD, thread delete, author fallback, quote matching) onto the new core/adapter, and let the old code keep running until the new path passes them.
  • Drop, don't port — much of the current suite tests removed behavior and should be deleted, not reimplemented: inline data-redline-anchor span materialization, PUT /api/document + whole-document serialize, update-html, repair-anchors, single-current-document assumptions, and quote + occurrence persistence. Accidentally resurrecting these is the main risk of "port the tests."

Suggested sequence:

  1. core (model + selector anchoring + fuzzy resolution + reconciliation) with tests.
  2. formats/html adapter (extractText / render / read+write block) with golden tests.
  3. app use-cases over core + adapter.
  4. server + agent on the new app layer; registry-based routing.
  5. src/client rebuild out of public/; remove public/app.js and the edit path.
  6. Build step + single-binary packaging.

Final gate: typecheck, unit tests, client build, and a browser smoke test pass before deleting the old client.

Test plan

  • Core / anchoring: selector capture; exact resolution; fuzzy re-anchor after edit-elsewhere, edit-inside-quote, and duplicate-created; orphan on wholesale rewrite; the three-state classification; self-healing hint refresh; reconciliation touches anchor only, never body.
  • HTML adapter (golden): read/write the embedded block only; content byte-identical outside the block; render passthrough + sanitization; state block removed when no threads remain.
  • Routes: every endpoint maps to an app command; response shapes incl. conflict + no-document; additive format field.
  • CLI: every command works against HTML fixtures and shares the app-command path with the server; registry routing (live server vs direct write).
  • Client: composer/replies/editing/delete-thread/active-thread/empty-state/save+conflict notices; highlight decoration over rendered DOM; orphaned/needs-review rendering.
  • Browser smoke: open a doc, select text, comment, reply, edit, delete a thread; externally edit the file (simulate the agent) and confirm anchors re-resolve, low-confidence flags show, and a destroyed region orphans gracefully.

Implementation knobs (decide in code)

Deliberately unspecified — pick sensible defaults during implementation and tune. None block the design.

  • Fuzzy confidence tiers. Thresholds for anchored vs needs-review vs orphaned. Start ~ exact or ≥0.9 = anchored, 0.6–0.9 = needs-review, <0.6 = orphaned; tune empirically.
  • Context window length. How much prefix/suffix to capture — default ~32 chars each side.
  • Fuzzy matcher. Vendor a small diff-match-patch–style matcher into core (no heavy dependency). Lock its behavior with fixture tests (known edits → expected anchored / needs-review / orphaned) before tuning thresholds, so matching doesn't become folklore.
  • docId wordlist. A curated, profanity-screened list (e.g. EFF); three words; regenerate on index collision.
  • Embedded schemaVersion. Bump it for the new selector model so the on-disk format is self-identifying.

Scope discipline

Get the HTML refactor right first. This rewrite ships a clean core + the single html adapter, the server/CLI/client on top, multiple-documents-per-server, and the binary packaging. Multiple documents is built at the server + API layer (the client ships a minimal document switcher to start; richer multi-document UX later). Multi-format support, user content editing (B), multi-client simultaneous editing, and live coordination are designed for (the seams exist) but not built here. Resist pulling those forward.
