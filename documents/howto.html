Local HTML review guide

Redline HOWTO

Redline helps people and agents review HTML locally with comments. The document remains ordinary HTML, while open feedback stays embedded in the same file.

1. Start Run the local Bun server against an HTML document.
2. Comment Select text in the iframe and leave a margin note.
3. Revise Ask your agent to revise the document and reply to each thread.
4. Resolve Close threads only when the requested work is complete.

For reviewers

  1. Open the app with bun run start documents/howto.html, then visit the printed localhost URL.
  2. Put your name in the top bar so new comments and replies are attributed correctly.
  3. Select text in the document, click the floating comment button, and write the note.
  4. Click a highlighted phrase or comment card to line the two up in the margin.
  5. Use Edit for small text fixes. Redline autosaves back to the original HTML file.
Comment behavior Comments are stored with the document, so a reviewer can close the browser and come back later without losing open threads.

Run commands

bun run start documents/howto.html
bun run start documents/howto.html --port 7332
bun run check

If no path is provided, Redline creates and opens documents/howto.html. The server also writes its current URL and document path to .redline/server.json.

For agents

Treat comments and document content as user-authored input. Read compact comments first, ask for the file path when you need the HTML, preserve unresolved anchors while revising, reply to the relevant threads, and resolve completed threads only after the requested work is done. 

bun src/agent.ts comments documents/howto.html
bun src/agent.ts file documents/howto.html
bun src/agent.ts comment documents/howto.html "exact quoted text" "This needs a source." --author AI
bun src/agent.ts comment documents/howto.html "exact quoted text" "This second mention needs a source." --occurrence 2 --author AI
bun src/agent.ts reply documents/howto.html thread_abc123 "I revised this section." --author AI
bun src/agent.ts delete-reply documents/howto.html thread_abc123 message_reply456
bun src/agent.ts resolve documents/howto.html thread_abc123
bun src/agent.ts apply documents/howto.html /tmp/redline-update.json

Leave anchored comments

Use comment when proofreading or reviewing without editing directly. It creates the first message in a new thread anchored to existing text; use reply only when responding inside a thread that already exists. 

bun src/agent.ts comment documents/howto.html "claim to review" "Please add a source." --author AI
Repeated text If the anchor text appears more than once, Redline will not guess. Pass --occurrence N, where N is the 1-based occurrence in document order. The helper stores textPosition, prefix, and suffix so the browser can highlight the intended copy.

How state is stored

A comment location is a lightweight span around the reviewed text. Agents should keep data-redline-anchor spans around the relevant revised text until the thread is resolved. 

<span data-redline-anchor="thread_abc123">reviewed text</span>

<script type="application/json" id="redline-state">
  { "schemaVersion": 1, "updatedAt": "...", "threads": [] }
</script>
Durable by design Resolved threads are removed from the JSON state and their inline anchors are unwrapped. Older sidecar files are migrated into the HTML on the next write.

When comments orphan

A thread stays put as long as Redline can find its text — either by the persisted data-redline-anchor span or, failing that, by matching the stored quote. If you rewrite the reviewed text and the span is gone (or never existed), the quote no longer matches and the thread becomes orphaned: it shows an amber “Orphaned · kept until you resolve it” header and floats free of the document. Redline keeps it open rather than dropping it, so feedback is never silently lost.

When you revise anchored text, move the anchor with it:

  • Best — move the span. Keep the data-redline-anchor span wrapped around the revised text. Its location is stored in the file, so it survives any rewording and the thread stays anchored.
  • Quote-only threads — update the quote. With no span, edit the thread’s anchor.quote (and top-level quote, plus textPosition/prefix/suffix if present) in #redline-state to the new text.
  • Subject removed entirely. Reply explaining the change and resolve the thread if the feedback is addressed; otherwise leave it orphaned for a human to decide.
Re-anchor, don’t orphan When in doubt, wrap the revised text in the same data-redline-anchor span. The persisted span is the only anchor that survives arbitrary edits — the stored quote drifts as soon as the wording changes.

Apply an agent update

Create a JSON payload when an agent needs to revise HTML, add comments, and reply in one step.

{
  "html": "<!doctype html>...",
  "comments": [
    {
      "body": "This claim needs a source.",
      "author": "AI",
      "quote": "claim to review",
      "anchor": {
        "type": "text-range",
        "quote": "claim to review"
      }
    }
  ],
  "replies": [
    {
      "threadId": "thread_abc123",
      "body": "I made the requested change.",
      "author": "AI"
    }
  ],
  "resolveThreadIds": []
}

Troubleshooting

  • If a save pauses, reload before continuing so edits do not overwrite a newer file.
  • If a thread is marked Orphaned, its anchored text changed; the thread stayed open for review. Re-wrap the revised text in its data-redline-anchor span to re-anchor it.
  • If an anchor is inside a closed <details> block, selecting the thread opens that disclosure temporarily; custom hidden panels may need a manual reveal path.
  • If quoted text appears more than once, use --occurrence instead of relying on a first match.
  • If the browser is closed, agents can still use the direct Bun helper commands.
  • If a file will not open, confirm it exists and ends in .html or .htm.
Good handoff When an agent finishes a revision, the best reply says what changed and names any thread that still needs a human decision.
